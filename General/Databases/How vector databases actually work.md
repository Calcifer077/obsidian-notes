Vector databases are what power the context in modern LLMs. 

A vector database indexes and stores vector embeddings for fast retrieval and similarity search, with capabilities like CRUD operations, metadata filtering, horizontal scaling, and serverless.

Embeddings (discussed below) are generated by AI models and have many attributes or features, making their representation challenging to manage. Vector databases offer optimized storage and querying capabilities for embeddings. Vector databases have the capabilities of a traditional database that are absent in standalone vector indexes and the specialization of dealing with vector embeddings, which traditional scalar-based databases lack.

With a vector database, we can add knowledge to our AIs, like semantic information retrieval, long-term memory, and more. The diagram below gives us a better understanding of the role of vector databases in this type of application:

![](../../assets/Pasted%20image%2020260630213448.png)

Let’s break this down:

1. First, we use the [embedding model](https://www.pinecone.io/learn/series/rag/embedding-models-rundown/) to create vector embeddings for the content we want to index.
2. The vector embedding is inserted into the vector database, with some reference to the original content the embedding was created from.
3. When the application issues a query, we use the same embedding model to create embeddings for the query and use those embeddings to query the database for _similar_ vector embeddings. As mentioned before, those similar embeddings are associated with the original content that was used to create them.

Below we have gone to the need of vector databases from ground up.
# Range Queries in 1 Dimension

What happens when we query an integer value in a database? 

Let's assume:

```sql
WHERE amount > 100;
```

Relational databases do this via B+ Trees. 

The basic idea behind the B+ Tree is that, at each node, we divide the space into two halves - a left and a right one.

The left one has all the values _less_ than the root, and the right half has all the values _more_ than the root.

If we think about the _dividing into two halves_, that is the main reason for B+ Trees efficiency, we're effectively reducing our search space in half at every single node.

# Range Queries in 2 Dimensions

Now, what happens if we try to apply the same logic in 2 dimensions?

Let's assume we have a food delivering app, we have coordinates, some latitude, longitude. Now, the algorithm wants to get all the free riders within 2 km radius of this restaurant so it can assign the order to someone.

If we were to go through the entire collection in the table, each available rider have its own coordinates, and than figuring out its location from the pickup and delivery location. _Just too much work and too slow._

	Can we use B+ Tree?

B+ Tree can only store one value, and we decide which half (left or right) to prune according to that. If we encode just the latitude, we would be blind or longitude and same goes for, if we encode the longitude.

So, we need to not just divide the surface into left/right, but also need a top and a bottom.

## QuadTrees

Quad - meaning 4, Trees.

As in a B+ tree, every insertion is based on whether it falls to left/right of a node; we do something similar in a QuadTree (but instead of 2, we divide the surface into 4).

### Idea
Imagine you have a square.

```
+---------+
|         |
|         |
|         |
|         |
+---------+
```

Split it into four equal parts:

```
+----+----+
| NW | NE |
+----+----+
| SW | SE |
+----+----+
```

Each of these four squares can be divided into four smaller squares if needed. 

### Structure of a Node

Typically each node stores:

```Java
class Node {
    boolean val;        // value of the region
    boolean isLeaf;     // whether this node is a leaf

    Node topLeft;
    Node topRight;
    Node bottomLeft;
    Node bottomRight;
}
```

A node can be either **Leaf Node** (represents an entire region having the same value) or an **internal node** (represents a mixed region).

### Examples

#### Example 1

```
1 1
1 1
```

Entire grid is uniform, so its a left node.

```
Leaf(1)
```

#### Example 2

```
1 1
0 0
```

The grid is not uniform, we need to split:

```
+---+---+
|1  |1  |
+---+---+
|0  |0  |
+---+---+
```

Tree:

```
	        Root
      /    |    |    \
    1      1    0     0
```

Each child is a leaf (uniform).

We have to recursive construction until every region is uniform or becomes a leaf.

Let our QuadTree have a surface of 100 X 100, currently empty.

Let's insert some points, (80, 80), (85, 85), (90, 90), (95,95):

As only one quadrant contains all the data, we will try to recursively divide it:

![](../../assets/Pasted%20image%2020260630083921.png)

Because the QuadTree splits based on where the centre is (half of available side length), rather than the data's median point, it can lead to heavy asymmetry:

- **Dense Clusters:** If you have 1000 points crammed into a tiny 1-meter square in Mumbai, the Quadtree will have to split dozens of times recursively to isolate those points into boxes that hold fewer than 4 points each. This creates a very deep tree branch.
- **Sparse Wastelands:** If you have 0 points in the middle of the Indian Ocean, that quadrant remains a single, giant, empty box that never splits.

**But why are we dividing in the middle?**

Because it's fast and easier to calculate.

But the production geospatial databases (like PostGIS in PostgreSQL) rarely use pure QuadTrees. Instead, they use **R-Trees** (Rectangle Trees).

## R Trees

It is like a spatial equivalent of a B-Tree. The main idea behind it is that instead of dividing space into fixed quadrants like a QuadTree, an R-Tree groups nearby objects into **Minimum Bounding Rectangles (MBRs).

MBR means that the nodes which are nearer to each other will be present in the same rectangle. 

Suppose we have six buildings:

```
A  B

C


          D
      E

              F
```

Group nearby ones:

```
Rect1
A B C

Rect2
D E F
```

Tree:

```
             Root
           /      \
       Rect1     Rect2
      / |  \     / | \
     A  B  C    D  E  F
```

Suppose we have to query something, we will start at the root, and check if the query intersect Rect1, if not entire subtree will be discarded.

When an R-Tree node reaches capacity:
1. It looks at the cluster of points. 
2. It finds a way to divide those points into two groups such that the total area of the two new wrapping rectangles is as small and tight as possible. 
3. The bounding boxes are allowed to overlap, and their shapes change dynamically based on the data

# Range Queries in N Dimensions

	Can we extend the idea of OctaTree to something like n(something) Tree?

But before trying to learn about n-dimensional space, let's visualize the n-dimensional objects.

As 3-D creatures we can't look beyond 3-D objects, so we will treat those objects as a **detailed profile or a feature checklist**.

These objects are called vectors.

Let's say we want to represent **Houses** as vectors.

**1D vector**
- Features: [ Size in sq ft ]
- Example: [ 2500 ]
- Visualization: A single point on a ruler.

**3D vector**
- Features: [ Size, Number of Bedrooms, Price ] 
- Example: [ 2500, 3, 500000 ] 
- Visualization: You can map this on an X, Y, Z graph.

**6D vector**
Now, let’s add more features. We want to capture: [ Size, Bedrooms, Price, Distance to metro (km), Age of building, Crime rate index ]. 
- Example House A: [ 2500, 3, 500000, 0.5, 5, 12 ] 
- Example House B: [ 2400, 3, 490000, 0.6, 6, 11 ]

If you look at the two arrays above, our brain instantly recognizes that House A and House B are _extremely similar_. They are basically identical twins in terms of their profile.

That is all an n-dimensional vector is. It's just a list of n quantitative metrices describing an object.

We all know how traditional databases work - they store strings, numbers, and other types of scalar data in rows and columns. On the other hand, a vector database operates on vectors, so the way it's optimized and queried is quite different. 

In traditional databases, we are usually querying for rows in the database where the value usually exactly matches our query. In vector databases, we apply a similarity metric to find a vector that is the most similar to our query.

A vector database uses a combination of different algorithms that all participate in Approximate Nearest Neighbour (ANN) search. These algorithms optimize the search through hashing, quantization, or graph-based search.

These algorithms are assembled into a pipeline that provides fast and accurate retrieval of the neighbours of a queried vector. Since the vector database provides approximate results, the main trade-offs we consider are between accuracy and speed. The more accurate the result, the slower the query will be. However, a good system can provide ultra-fast search with near-perfect accuracy.

Here's a common pipeline for a vector database:

![](../../assets/Pasted%20image%2020260630214347.png)

1. **Indexing**: The vector database indexes vectors using an algorithm such as PQ, LSH, or HNSW (more on these below). This step maps the vectors to a data structure that will enable faster searching.
2. **Querying**: The vector database compares the indexed query vector to the indexed vectors in the dataset to find the nearest neighbors (applying a similarity metric used by that index)
3. **Post Processing**: In some cases, the vector database retrieves the final nearest neighbors from the dataset and post-processes them to return the final results. This step can include re-ranking the nearest neighbors using a different similarity measure. The vector database indexes vectors using an algorithm such as PQ, LSH, or HNSW. It just creates n-dimensional array of fed documents. It maps the vectors to a data structure that will enable faster searching.

In reality, AI models use 1,536 or 3072 dimensions. We don't know exactly what feature each dimension represents because the AI figured them out on its own, but the underlying principle is the same.

If we try to use the same concept as a QuadTree (2$^2$ = 4 children) and an OctaTree (2$^3$ = 8 children), for a 100-dimensional vector we would have 2<sup>100</sup> nodes.

A single node division in this case would instantly crash any computer's memory.

We needed something else to solve the same problem that we did easily in a lower dimension - _"how to find the nearest neighbour in higher dimensions?"_

## Social Network Graph

There is a concept which states that:

	Every human on Each is connected by at max 'n' degrees of separation.

If we want to contact a person, we obviously don’t have a global map of every person on Earth with their coordinates; instead, we have these _connections_.

If you want to pass a message to a random software engineer in Kenya, you don’t have to search all 8 billion people sequentially. 
1. You might ask your friend who works at a global tech company. 
2. They introduce you to their colleague in London. 
3. That colleague knows someone managing a remote team in East Africa. 
4. And… you’ve reached your destination in 4 jumps.

This works because human society is a **Small World Network**. It has two distinct types of connections:

- **Local Clusters**: You have a close circle of friends who all know each other (high clustering)
- **Global Hubs**: You or your friends know a few people who live far away, act as bridges to completely different social circles, or travel frequently.

## Transforming Vectors into a Social Network Graph 

But how do we transform these mathematical constructs (vectors) into something that can simulate human connections? 

Well, instead of thinking of vectors as being drawn on a 1000-dimensional space, let’s think of it as a person living on that coordinate. 

When we insert a vector, the database connects it to its “nearest neighbours” with explicit edges, turning the database into a massive graph. 

Now, to find the nearest neighbour of a vector, we just follow the nodes and compare distances until we can’t find any nodes closer.

But if we build one massive graph of interconnectivity between people, it would be extremely difficult to find a node from one side of the graph to another.

To fix this bottleneck, an algorithm called _Hierarchical Navigable Small Work (HNSW)_ was designed. 

HNSW introduces a hierarchical multi-layered graph, something like a skip-list.

![](../../assets/Pasted%20image%2020260630094021.png)

	A probabiltiy skip list structure, we start on the top layer. If our current key is greater than the key we are searching for (or we reached the end), we drop to the next layer.

Let's understand how a search operation happens in this space.

### Layer 2 - The Airplane Layer

This layer contains only a tiny fraction of your vectors (the “global hubs”). 
- The connections here span massive distances across the 1000D space. 
- _The Search_: When a query comes in, the database starts at a random node on this top layer. It takes massive, sweeping jumps across the space to quickly get into the correct general timezone/continent of your query. Once it can’t find any closer nodes on this layer, it drops down a level.

### Layer 1 - The Highway Layer

This layer contains a few more vectors. The connections are shorter, bridging cities or neighbourhoods. 
- _The Search_: The database picks up exactly where it dropped off from Layer 2. It zooms in closer to the target vector, hopping between regional nodes until it can't get any closer. Then, it drops down to the final layer.
### Layer 0 - The Local Streets

This layer contain **all of our vectors**. Every single data point is mapped here with its absolute closest semantic neighbours.
- _The Search_: The database lands right in the exact neighbourhood of your target. It does a quick, short crawl through the immediate local connections to find the exact nearest matching vectors.

![](../../assets/Pasted%20image%2020260630204421.png)

If we compare it to our lower-dimensional Quad/Octa Tree -
- **No Spatial Explosion**: The database never creates 2<sup>100</sup> empty boxes. If you have 10000 vectors, you have exactly 10000 nodes in your graph, plus a few duplicate hub nodes on upper layers. Space is completely determined by where the data _actually_ exists.
- **Amazing Speed**: Because you can use the top layers to skip over massive expanses of irrelevant data instantly, the time it takes to find your nearest neighbour scales logarithmically, O(log N)—even if your vectors have thousands of dimensions.

How do we insert a node in this graph?

## Insertion in HNSW

Let's try to insert a brand new vector in this database. Let's call it a node.

### Step 1 - Assigning a Random Max layer

Since probability goes downwards layer by layer. **100%** of nodes get inserted into Layer 0 (the local streets). Roughly **10%** make it to Layer 1, **1%** to Layer 2, and so on.

We get a random number between 0 and 1. Let's say our New Node gets lucky and its max layer is determined to be **Layer 1 (the highway layer)**. This means it will exist on both Layer 0 and Layer 1.

### Step 2 - Downward Search (Finding the neighbours)

The database enters the graph at the top-most layer, Layer 2 (the Airplane Layer), at a default entry point.

1. On **Layer 2**, it performs a quick search to find the node that is closest to our New Node.
2. Once it finds that closest node, it drops down to **Layer 1** at that same coordinate.

### Step 3 - Inserting into Different Layers 

Now that the database has dropped down to Layer 1 (Highway Layer, the maximum layer assigned to our New Node), the real insertion work starts - 
1. **Find the M closest Friends**: On Layer 1, the database searches for the absolute closest neighbours to our New Node. The database has a configurable parameter called M (e.g., M=4), which is the maximum number of connections any node can have at a particular level. It finds the top M closest neighbours.
2. **Bi-directional Linking**: The database places the New Node onto Layer 1 and draws lines (edges) connecting it to those M neighbours. These connections are bi-directional. The neighbours now point to the New Node, and the New Node points to them (so that we can reach both of them vice versa).
3. **Drop and Repeat**: The database drops down to Layer 0, takes the closest points it found from Layer 1 as its starting point, searches for the closest M neighbours on the crowded local streets, and connects itself to them as well.

![](../../assets/Pasted%20image%2020260630210038.png)

If you pay close attention to the algorithm above, you might find that there’s a huge problem with this approach. 

What if a few nodes are really popular? They might be connected to a lot of neighbours, making them highly inefficient to narrow down our searches to lower layers. 

To prevent this, HNSW runs a pruning algorithm during insertion (you can draw the similarities of rebalancing of B+ Trees, or rearranging the points after division of the surface in a QuadTree).

## Pruning of Neighbours

If adding the New Node causes an existing node to exceed its limit of M connections, that old node is forced to re-evaluate its friends:

- It looks at its M + 1 connections.
- It measures the distance to all of them.
- It keeps the closest/most structurally diverse ones and breaks the connection with the furthest one.

This keeps the density of the entire network uniform and bounded, ensuring that no single node ever becomes a bottleneck. 

We have essentially built the algorithm that most vector databases use internally to understand the context before giving it to an LLM.

# How it connects to RAG (Retrieval Augmentation Generation)

The algorithm we just figured out is the R (retrieval) part of the RAG.

It is essential to know how to find useful contexts in a huge ocean of interlinked information. 

You could try to do RAG without a vector database by just stuffing every file you own into the LLM prompt every single time. But LLM prompts have strict context windows, and passing millions of words to an LLM for a simple question is highly expensive and absolutely slow. 

The vector database acts as the intelligent filter. It reduces gigabytes or terabytes of corporate knowledge down to the exact 3 or 4 paragraphs needed to answer a specific question in under 50 milliseconds.
